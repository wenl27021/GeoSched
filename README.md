# `GeoSched.py` 代码总结与分析

## 1. 整体功能概述

`test1.py` 是一个高级的卫星任务调度比较与分析平台。其核心目标是解决一个复杂的多约束、多目标的调度问题：将一系列具有不同优先级、时间窗口和资源需求的观测任务，分配给一组具有不同能力和资源限制的卫星。

该脚本不仅执行任务分配，更重要的是，它**实现、执行并系统性地比较了四种不同复杂度与智能水平的调度算法**，旨在评估哪种算法在满足物理约束（如能量、时间窗口）的前提下，能够最有效地实现以下两个核心业务目标：

- **最大化总任务收益**
- **最大化总完成任务数**

最终，系统会为每种算法生成详细的日志和性能统计数据，并进行横向对比，为决策者提供数据支持。

---

## 2. 代码架构与工作流程

代码采用单文件结构，但逻辑上非常清晰，遵循了一个“**数据加载 -> 模型训练 -> 多策略调度 -> 结果分析**”的完整工作流。

#### 工作流程详解：

1.  **初始化阶段**:
    *   程序从 `satellites_20.csv` 和 `tasks_500.csv` 读取原始数据。
    *   `load_satellites` 和 `get_all_unique_task_types` 函数将这些数据转换为程序内部使用的 `Satellite` 和 `Task` 数据结构。

2.  **GNN/ASGA 模型训练阶段**:
    *   这是代码中最具前瞻性的部分。它通过 `build_graph` 将所有卫星和任务构建成一个图结构。
    *   然后，它实例化一个 `ASGAT_GNN` 模型（一个基于自适应图注意力的神经网络），并调用 `train_gnn_model` 对其进行**训练**。
    *   训练过程是一个**自监督学习**任务：模型通过预测图中节点之间是否存在连接来学习节点的特征表示（嵌入），从而让模型理解卫星与任务之间的内在关联。

3.  **多算法调度与评估阶段**:
    *   脚本采用严谨的比较方法：通过 `copy.deepcopy` 为每种调度算法创建一个完全独立的“沙箱”环境，确保比较的公平性。
    *   **三种基准/启发式算法** (`plain_schedule`, `greedy_schedule`, `greedy_schedule_agent`) 被依次调用。
    *   **最高级的 `multi_objective_schedule`** 被调用，并将**训练好的 `ASGAT_GNN` 模型**作为参数传入，利用其生成的嵌入向量来指导优化过程。

4.  **结果汇总与输出**:
    *   每种算法执行完毕后，`print_satellite_statistics` 会立即计算并输出其性能指标到对应的日志文件和控制台。
    *   所有结果被收集并在程序末尾输出一个最终的、跨所有算法的性能总览，方便直观比较。

## 3. 核心算法剖析：从基础规则到混合智能

脚本中的四种调度算法在复杂性和智能程度上层层递进，构成了一个完整的算法演进阶梯。所有算法都构建在一系列基础可行性检查之上。

### a. 基础模块：可行性检查 (`is_feasible`)

这是所有调度算法的基石。在尝试为任务分配卫星之前，必须通过 `is_feasible` 函数进行验证。该函数捆绑了五个独立的检查，确保分配在物理上是可能的：

1.  **状态检查 (`check_satellite_status`)**: 卫星是否处于“服务中”状态？
2.  **窗口检查 (`check_windows_today`)**: 卫星当天是否还有可用的观测窗口？
3.  **类型支持检查 (`check_task_type_support`)**: 卫星是否支持该任务的类型？
4.  **时间冲突检查 (`check_time_conflict`)**: 任务的执行时间段是否与卫星上已安排的其他任务重叠？
5.  **能量检查 (`check_energy_available`)**: 卫星的剩余能量是否足够执行该任务？

只有当所有这些检查都通过时，一个任务才能被考虑分配给该卫星。

### b. 算法一：顺序调度 (`plain_schedule`)

这是最简单的基准算法，其逻辑如同其名：

- **工作方式**: 按任务在 `tasks_500.csv` 文件中的原始顺序，逐一为它们寻找可用的卫星。对于每个任务，它会遍历所有卫星，一旦找到第一个满足 `is_feasible` 条件的卫星，就立即进行分配，然后不再为该任务寻找其他可能性。
- **特点**: 简单、快速、但非常短视。它完全不考虑任务的优先级或收益，也无法进行任何优化决策，调度结果的好坏在很大程度上取决于任务和卫星在输入文件中的排列顺序。

### c. 算法二与三：带替换机制的贪心调度 (`greedy_schedule` & `greedy_schedule_agent`)

这两个算法引入了“贪心”策略，代表了启发式优化的核心思想。它们不再是简单地接受第一个可行的方案，而是试图做出局部最优的选择。

- **核心思想**: 高价值的任务应该被优先处理。因此，算法首先将所有待调度任务按照 **优先级（高到低）** 和 **收益（高到低）** 进行排序。

- **关键机制：带替换的分配 (`try_assign_with_replacement`)**
  - 这是贪心算法的“智能”所在。当一个高优先级的新任务因为时间冲突而无法直接分配给某个卫星时，该函数会检查是否可以“踢掉”一个或多个已经占位的、但优先级较低的旧任务。
  - 替换的可行性取决于两个条件：1) 新任务的优先级必须高于所有与之冲突的旧任务；2) 将所有冲突的旧任务移除后，“退还”的能量加上卫星的剩余能量，必须足够执行新任务。

- **两种贪心策略的差异**:
  - `greedy_schedule` (简单贪心): 当有多个卫星都可以通过替换来接纳新任务时，它会选择一个**导致最少任务被替换**的方案。其目标是尽可能减少调度的“颠簸”。
  - `greedy_schedule_agent` (智能贪心): 在前者的基础上增加了一个决策维度。如果多个卫星方案的替换任务数相同，它会进一步计算并选择那个能带来**最大净收益增益**（`新任务收益 - 被替换任务总收益`）的方案。这使得它在局部决策上更具“商业头脑”。

### d. 算法四：基于GNN与进化算法的多目标调度 (`multi_objective_schedule`)

这是整个脚本中最先进、最复杂的混合智能算法。它将调度问题提升到了一个全新的维度，不再是逐个任务做决策，而是将所有任务的分配作为一个整体进行全局优化。

- **高层理念**: 将任务调度视为一个**多目标优化问题**，并使用强大的 **NSGA-III 进化算法** 在庞大的解空间中搜索一组最优的“权衡”解（Pareto最优解集）。这里的两个目标是：1) 最大化总收益；2) 最大化总任务数。

- **核心组件一：自适应图注意力网络 (ASGA-GNN)**
  - **角色**: ASGA-GNN 的作用是**“理解”问题**。它将卫星和任务表示为图中的节点，节点的属性（如能量、收益、轨道参数）作为特征。然后，通过图神经网络学习这些节点的高维特征表示（嵌入）。
  - **自适应注意力 (`ASGATLayer`)**: 与普通GNN不同，该模型使用的注意力层可以动态地学习图中节点之间的关系强度。例如，模型可以学到一个高能量消耗的任务与一个高能量储备的卫星之间有更强的“注意力”。
  - **自监督训练**: 这是画龙点睛之笔。模型在调度开始前会进行一个短暂的**自监督训练**。它通过尝试重构图的结构（预测哪些节点之间应该有连接）来学习。这个过程迫使模型生成的嵌入向量不仅包含节点的自身属性，还蕴含了它在整个问题网络中的“角色”和“关系”信息。

- **核心组件二：NSGA-III 进化算法**
  - **角色**: NSGA-III 的作用是**“解决”问题**。它是一个强大的搜索引擎，能够在巨大的可能分配方案中找到最优解。
  - **GNN赋能的初始化 (`gnn_based_sampling`)**: 这是GNN与进化算法的**关键连接点**。NSGA-III的搜索不是从完全随机的分配方案开始的，而是由训练好的GNN模型来“指导”其第一步。它通过计算任务和卫星嵌入向量之间的**余弦相似度**来评估它们的“匹配程度”，并优先生成那些匹配度高的初始分配方案。这相当于给一位经验丰富的专家（NSGA-III）一份高质量的“草稿”（GNN的建议），极大地提高了搜索效率和最终解的质量。
  - **进化与选择**: 在获得高质量的初始种群后，NSGA-III通过多代的交叉、变异等操作，不断迭代、优化调度方案，最终收敛到一组无法再被改进的Pareto最优解。脚本最后从中选择总收益最高的一个作为最终执行的方案。

## 4. 如何运行

1.  **环境配置**: 确保已安装所有必需的Python库。可以将以下内容保存到 `requirements.txt` 文件中，然后运行 `pip install -r requirements.txt`：
    ```
    torch
    torch-geometric
    numpy
    pymoo
    ```

2.  **文件准备**: 确保 `test1.py`, `satellites_20.csv`, 和 `tasks_500.csv` 文件位于同一个目录下。

3.  **执行脚本**: 在终端中运行以下命令：
    ```bash
    python test1.py
    ```

4.  **查看结果**: 
    *   脚本运行期间，GNN模型的训练进度和每个调度算法的实时统计信息将打印在控制台。
    *   运行结束后，一个名为 `output_logs` 的文件夹将被创建，其中包含了每个算法的详细日志和最终的性能对比总结 `summary_log.txt`。
